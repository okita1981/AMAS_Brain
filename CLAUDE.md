# AMAS のコンセプトとコア（最重要・全判断の基準）

## 誰のための何か
デジタル広告をこれまで導入できなかった人のためのサービス。
予算が限られ代理店に相手にされない、マーケ人材がいない人たち——
町の塗装屋、士業、飲食店の求人など。
代理店主導でもインハウス化が進んでも埋まらない「マーケ難民」のポジションを取る。

## 最優先の価値基準：UI/UX ＞ 精度
このサービスの成否は配信精度ではなく、素人が直感的に使えるかで決まる。
あらゆる実装・仕様判断の基準は「マーケ知識ゼロの個人事業主が、迷わず操作できるか」。
精度の作り込みより、操作のわかりやすさ・少額導線・直感的な成果表示を優先する。

## コア3点 ＝「今すぐ・誰でも・少額で」

### ① 今すぐ：思いついた瞬間に広告を出せる（スピード）
「広告出したい → 今日出せる」までの距離を極限まで短くする。
広告業界はここが異常に重い（見積・打ち合わせ・専門家が必須）。
AMASは 見積不要・打ち合わせ不要・専門家不要 で "広告出したい→出せた" を最短にする。

### ② 誰でも：マーケ知識がなくても成果が理解できる（わかりやすさ）
CTR / CVR / CPA / ROAS といった指標は提供する（上級者やデータを見たい人のために残す）。
ただし、それらを理解しなくても成果がわかる作りを最優先する。
ユーザーが本当に知りたいのは「問い合わせ来た？ 応募来た？ 電話鳴った？」。
専門用語を消すのではなく、専門用語を知らなくても直感的に成果が伝わる
表現・導線を上に重ねる。広告の専門家にならなくていい、がコア。

### ③ 少額で：小さく始められる（心理的ハードルの低さ）
代理店は月20〜50万が普通。だが町の塗装屋は「まず1万円試したい」。
"失敗しても痛くない" が価値。少額チャージ・少額出稿を当たり前にする。

## 実装時の判断指針
- 新機能・仕様変更は、まず「コアを磨くものか、枝葉か」を問う。
- コアの直感性を損なう複雑さは、高機能でも採用しない。
- 仕様変更を厭わず、UI/UX の追求を繰り返す前提で開発する。
- 迷ったら「これは"今すぐ・誰でも・少額で"を強めるか？」に立ち返る。

# ウォレット（財布）設計 ＝ サブスクとチャージを完全分離（コンセプトに次ぐ重要設計）

## 大原則：2つの財布は性質が違う。混ぜない。
ユーザーのお金は性質の異なる「2つの財布」に分け、データ上も画面上も分離する。

### サイフA：サブスク（＝AMASの売上）
- 性質：AMASというツールの月額利用料。AMASの課税売上。
- 課金：Stripe のサブスクリプション（月額自動課金）。
- 消費税：課税売上なので消費税がかかる（AMASが納税）。
- 料金プラン：
  - 無料プラン：残す（持ち出しで提供）。「少額で・誰でも」の入口を守るため。
  - ライト：9,800円/月（広告費チャージ上限 10万円/月）
  - スタンダード：29,800円/月（広告費チャージ上限 30万円/月）

### サイフB：チャージ（＝広告費の原資＝預り金）
- 性質：ユーザーが広告を出すための原資。AMASの売上ではない（預り金・立替金）。
  AMASは広告費にマージンを一切乗せない（媒体費は素通し）。
- 消費税：媒体（Google/Meta）への支払い時に消費税がかかるため、
  チャージから税分を裏で区分して確保する。表示はシンプルに（残高だけ見せる）。
- チャージ上限：プランに連動（ライト10万/月、スタンダード30万/月）。
  → この上限が「手数料吸収の上限」も兼ね、損失の青天井化を防ぐ安全弁になる。

## 課金サイクル（確定）
- 契約日基準。ユーザーが契約した日を起点に毎月その日でリセット
  （例：5日契約なら毎月5日更新・5日リセット）。Stripeサブスクの標準方式に合わせる。

## チャージ上限は「残高連動」（確定・青天井防止の核心）
- 今月チャージできる額 ＝ プランのチャージ上限 − 現在の残高。
  例（ライト上限10万）：残高5万なら今月はあと5万までチャージ可。残高10万なら今月はチャージ不可。残高0なら満額10万チャージ可。
- これにより「残高 ＋ チャージ可能額 ≦ プラン上限」が常に成立し、
  保有残高がプラン上限（ライト10万/スタンダード30万）を超えて積み上がらない。
  → 青天井化を構造的に防止し、預り金の積み上がり（資金決済法リスク）も抑制する。
- 残高は繰り越す（月をまたいでも消えない）。コア「失敗しても痛くない」を守る。
  ただし上限連動のため積み上がりすぎない。

## チャージの絶対条件：入れた額が満額そのまま広告費になる
- ユーザーがチャージした金額は、満額そのまま広告費に使える。
  決済手数料は AMASが吸収する（ユーザーに手数料を意識させない）。
  → コア「少額で・誰でも」を守るための絶対条件。
- 手数料吸収の原資はサブスク（サイフA）で賄う。
  試算：ライト9,800円に対し上限10万を全額クレジット(3.6%)でも手数料は3,600円で十分ペイ。
  スタンダード29,800円も同様に十分ペイ。

## チャージ手段（ユーザーが選べる。どれでも満額広告費に使える）
- PayPay（QR）：手数料 約1.6〜1.98%。スマホでピッ。一番手軽でターゲット層と相性良。メイン導線候補。
- 銀行振込（Stripe経由）：手数料 約1.5%。最安。まとまった額のチャージ向け。
- クレジットカード：手数料 約3.6%。一番高いが、慣れている人向けに残す。

## 返金（確定）
- チャージ残高の返金には応じない。
- 記載場所：Wallet画面には大きく出さない（不安を煽るため）。ヘルプに正式記載。
- 【要確認】返金不可は重要事項のため、利用規約にも明記し、課金・チャージ前に
  ユーザーが同意する形にする必要がある（弁護士確認）。ヘルプ記載だけでは不十分。

## 未解決・要専門家確認（本ローンチ前に必須）
- 【最重要】資金決済法：預り金（チャージしてためる）モデルは「前払式支払手段」等の
  規制対象になる可能性。供託金・届出義務が生じうる。→ 弁護士・行政書士に早急に確認。
- 税理士確認：チャージを「預り金・立替金」として処理する妥当性、立替の実態を証拠として
  残す運用、インボイス対応。
- PayPay：オンライン決済（SaaSへの組み込み）の対応可否・審査要件をPayPayに確認。
- PayPayマイストア ライトプラン（月1,980円で手数料1.98%→1.60%）への加入是非は
  チャージ総額の見込みで判断（運用開始後のチューニング）。

## 実装時の注意（既存コードとの関係）
- 既存実装では Stripe顧客IDは /users/{uid}、残高は /wallets/{uid}（balance_total /
  balance_ad_budget=税抜 / tax_holding=消費税分）に分離済み。この税区分の思想は維持する。
- ただし現状UIは「既定¥55,000・税務用語・12ステップ」で、上記の設計と乖離している。
  Wallet画面はこの設計に沿って「少額・シンプル・3手段から選択」へ作り直す必要がある（次の課題）。
