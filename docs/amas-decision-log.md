## D1｜恒久的ターゲット

**決定済み**:
- AMASの恒久的な主対象は、**広告運用の専門知識・人材を持たない小規模事業者**。
- 代理店は主対象ではない。Agencyプランが存在する場合も、販売パートナー・導入支援者・複数顧客管理などの**補助的位置づけ**とする。
- AMASを代理店業務効率化プロダクトへ転換しない。

**決定理由**:
Fable Reviewが検出した資料間の主軸未決（コンセプト文書は「小規模事業者の民主化」、収益シミュレーションはAgencyプラン・月額¥198,000を初動最重要ターゲットと記述）を解消する必要があったため（Evidence §3・§16-2）。AMASの存在意義（CLAUDE.md冒頭のコンセプト「マーケ難民」）と整合させ、収益計画側を主対象の定義から切り離した。

**不採用案**: Agency（代理店）主軸化。収益シミュレーション文書が示唆していた方向。理由: 採用するとAMASが「広告運用の民主化」という中心コンセプトから逸脱し、一般的な代理店向け業務効率化SaaSに縮小するため。

**影響範囲**: Product Charter §2・§6（Non-Goals）、Architecture ReviewのUX設計判断全般、Master RoadmapのPhase設計（Agency向け機能は現行Phaseに含めない）。

**再検討条件**: 小規模事業者向け提供が実証された後、Agencyを補助チャネルとして拡張する場合のみ再検討。主対象の変更自体は再検討しない（恒久方針）。

## D2｜中心価値

**決定済み**:
AMASの価値の源泉は、ユーザーが広告媒体の知識を持たなくても、

- AMASへ登録
- 自社情報またはURLを入力
- 広告の目的を選択
- 予算をチャージ
- AMASの提案を確認・承認

するだけで広告を開始・運用できること。媒体別の複雑性はユーザーへ戻さず、AMASの裏側で吸収する。

**決定理由**: CLAUDE.md冒頭のコンセプト（「今すぐ・誰でも・少額で」）をGate 2で正式な中心価値として再確認・正本化。Fable Reviewが指摘した「UIは一気通貫で動くが、承認以降がすべて演出」という現状（Evidence §8）を踏まえ、承認までのUX契約を明文化した。

**不採用案**: なし（既存コンセプトの正本化であり、対立する代替案は検討対象外）。

**影響範囲**: Product Charter §7（UX原則）、Architecture Review §7（承認境界）、Master RoadmapのPhase 1完了条件（D8のE2Eループが本UX契約を満たすこと）。

**再検討条件**: なし（恒久方針）。ただし「AMASの裏側で吸収する」複雑性の範囲は媒体追加ごとにArchitecture Reviewで具体化する。

## D3｜広告アカウントモデル

**決定済み**:
標準体験として、ユーザー自身にGoogle AdsやMeta等の広告アカウント開設・OAuth接続・支払い設定を求めない。AMAS側で管理する広告アカウント構造を通じて運用する。表側の体験は統一する。

**未決（Open・Architecture Reviewで継続検討）**:
実際のアカウント所有・管理方式（子アカウント発行型／ユーザー所有＋権限委譲型／その他）は、媒体ごとに規約・リスク・移管・停止・違反波及を検証してから設計する。裏側は媒体別に最適化してよい。**この具体的方式は本エントリ時点では未確定**。

**誤読防止の明記（R-07）**: 裏側の媒体別実現方式として複数の技術候補（子アカウント発行型・ユーザー所有＋権限委譲型・その他）を比較検討することは、**決定済み欄の標準UX（ユーザーに広告アカウント開設・OAuth接続・媒体側支払い設定を要求しない）を再オープンするものではない**。これらの候補はあくまで「標準UXを満たすための裏側の実現方法の選択肢」として比較されるものであり、いずれの候補が採用されても、ユーザー自身がOAuth接続等の操作を行うことを標準体験に持ち込むことはない。特に「ユーザー所有＋権限委譲型」という候補名は、内部的な権利構造（誰が媒体アカウントの権利を保持するか）を指す技術的検討事項であり、**ユーザー自身にOAuth接続操作を求める標準体験を意味しない**。標準体験としてユーザー操作を求めない、という決定済み事項は不変である。

**決定理由**: Fable Reviewが検出した資料間の設計反転（AMAS利用規約第4条・営業デッキ・概念正本は「子アカウント自動発行」を前提とする一方、`アカウント問題.docx`（2026-04-03）は同モデルが規約違反時の連鎖停止等のリスクを持つと認定し、OAuthユーザー所有モデルへの移行を「サービス持続の必須要件」と結論していた。Evidence §2・§16-3）。この矛盾を放置せず、**ユーザー体験としての方針**（表側は統一・ユーザーにOAuth接続を要求しない）だけを確定し、**実装方式**（裏側の権利構造）は検証未了のまま未決と明記することで、誤った断定を避けた。

**不採用案**: 子アカウント自動発行モデルをそのまま踏襲すること。理由: `アカウント問題.docx`が認定した規約リスク（ビジネスオーナー不一致・違反時の連鎖停止・身元確認書類の矛盾）を放置することになるため。

**影響範囲**: Architecture Review §6（媒体Adapter）に「オープンな設計課題」として明記。Security and Wallet Design §2（認可）。Master RoadmapではPhase 1（Google Ads・PAUSED入稿のテストアカウント）は本決定の影響を受けない小規模な範囲で先行実証できるが、Phase 5（Multi-platform Expansion）までにはアカウント所有方式の確定が必要。

**再検討条件**: Google Ads側の検証（Phase 1〜2の中で）が完了し、媒体ごとの規約・リスクが整理された時点で、具体的な所有・管理方式を確定する。

## D4｜収益・Walletモデル

**決定済み**:
- AMAS利用料は月額サブスクリプション。Walletチャージは広告費の原資。両者を完全に分離する。
- Wallet前払い方式はAMASの恒久的な基本体験として維持する。
- ユーザー表示は会計用語より「実際に広告配信へ使用できる金額」を優先する。表示例:
  ```
  チャージ額：10,000円
  広告配信に使える金額：9,091円
  消費税相当額：909円
  ※消費税相当額は広告配信には使用されません。
  ```

**未決（Open・専門家確認事項。Commercial / Financial Go Gate前に確定が必須）**:
- 税相当額の内部会計処理、媒体請求との関係、端数処理、インボイス対応
- 預り金モデルの資金決済法・資金移動業該当性

**決定理由**: CLAUDE.md既存設計（サイフA/サイフBの分離、税区分）を恒久方針として再確認。Fable Reviewが指摘した「チャージ上限・残高連動がサーバー側に不在」（Evidence §9、P1-5）という実装欠落は方針自体の否定材料ではなく、**再構築で埋めるべき欠落**として扱う（Architecture Review・Security and Wallet Design側の課題とする）。表示のシンプルさを損なわない前提を明記したのは、専門家確認（税務・法務）の結果が複雑な会計処理を要求しても、ユーザー向けUXを後退させないため。

**不採用案**: サブスクとWalletを統合した単一課金モデル。理由: CLAUDE.md原設計の「2つの財布は性質が違う。混ぜない」という大原則、および預り金の会計上の独立性（マージン0%・媒体費素通し）という中心価値の一部を損なうため。

**影響範囲**: Product Charter §8（収益構造）。Security and Wallet Design §4〜9（Wallet台帳・チャージ・税相当額表示・予約・媒体消化・返金）。Master RoadmapのCommercial / Financial Go Gate（専門家確認の必須項目として列挙）。

**再検討条件**: 専門家確認（税務・資金決済法）の結果、表示例の会計的整合性に問題が見つかった場合、内部処理のみ調整しUXは維持する方向で再検討する。

## D5｜CVI

**決定済み**:
- CVIの思想は維持するが、**実データで算出可能になるまでは内部研究指標とする**。
- **未定義・乱数のCVIをユーザー向け主要KPIとして表示しない。**
- 当面の最適化は、広告目的に応じた実測KPI（広告消化額・CV数・CPA・売上・ROAS・予約数・問い合わせ数等）を使用する。

**未決（再検討条件つき）**:
CVIの正式定義（定義式・入力データ経路）と、ユーザー向け上位指標への再昇格は、利益・成約日数などの入力経路が成立した後に再検討する。

**決定理由**: Fable Reviewが検出した「CVIは『唯一の目的』と位置づけられながら、定義式・計算実装・入力データ経路のすべてが存在せず、資料自身が『現在はシミュレーション値』と自認している」（Evidence §3・§7、P1-3）という事実に基づく。未定義・乱数の指標を主要KPIとして提示し続けることは、Fable ReviewのP1-3（実データと区別できない演出）と同種のユーザー誤認リスクを再生産するため、格下げを決定した。

**不採用案**: 
1. CVIを現状のまま（乱数演出込みで）主要KPIとして維持する — ユーザー誤認・信頼毀損リスクのため不採用。
2. CVIを完全に廃止する — コンセプトの中核であり、実データが揃った後に再評価する余地を残すため不採用（格下げに留める）。

**影響範囲**: Product Charter §6（Non-Goals：CVIを主要KPIにしない）。Architecture Review（CVIを内部研究指標として扱う設計、乱数加算ロジックの廃止＝D6の廃止対象と連動）。Master Roadmap（CVI正式定義はParking Lot、再検討条件は本エントリを参照）。

**再検討条件**: 利益・成約日数（またはそれに相当するデータ）の入力経路がAMAS上で成立し、実データからCVIを算出できるようになった時点で、正式定義・上位指標化を再検討する。

## D6｜再構築方針

**決定済み**（Gate 2で新たに決定した独立方針。Fable Review §13の案A〜Dのいずれか単独の採用ではない）:

- 現行AMASへの継ぎ足し修正を本線にしない。
- AMASのコンセプト、ブランド、主要UX、AI生成資産を維持する。
- 実行アーキテクチャは再構築する。
- Switch Mediaから現時点で取り込むのは、業務知識と設計パターン。
- Google Ads／PAUSED入稿／人間承認必須の最小E2Eを先に実証する（詳細はD8）。
- 共通コアは先に作らない。
- AMASとSwitch Media双方で、同じ責務・データ契約・ライフサイクルが実証された後に、共通コア抽出の可否を判断する。

この方針を要約すると：**「AMAS実行アーキテクチャ再構築＋最小E2E実証。共通コア化は実証後の条件付き判断」**。

**決定理由**: Fable Review §13は案A（段階的改善）・案B（フル再構築）・案C（Switch Media基盤を共通コアに抽出）・案D（最小E2E実証先行）の4案を提示し、案C＋案Dの組み合わせを推奨していた。Gate 2ではこの推奨を単純採用せず、以下の理由で独立方針として決定した:
- 案Cの「共通コアを先に抽出する」は、Switch Media側（ad-campaign-studio）自身がcross-repo importという密結合を抱え、CSV出力ビルダーに「媒体別情報が反映されない」既知の欠陥を持つ（Evidence §11）。両者の責務・データ契約・ライフサイクルが実証される前に共通コアを抽出すると、AMAS側にSwitch Media側の未解決の欠陥を持ち込むリスクがある。
- 案Bの「フル再構築ですべて捨てる」は、AI生成資産・主要UXという実際に価値のある資産までも喪失させるため過剰。
- したがって、AMASの資産を残しつつ実行アーキテクチャのみを再構築し、Switch Mediaからは「コード」ではなく「業務知識と設計パターン」のみを移植し、共通コア化の判断は両者の実証結果を待って行うことにした。

**不採用案**:
1. **案A（現行AMASへの継ぎ足し修正）** — 単独では不採用。無認証API・偽装実行パス・単一巨大コンポーネントという構造的欠陥の上に実配信を接続することになり、安全性が低いため（Evidence §13案A評価）。
2. **フルスクラッチ（すべて捨てる案）** — 不採用。AI生成プロンプト群・UXデザインという価値のある資産まで喪失するため。
3. **共通コアの先行抽出（Switch Media基盤を最初から共通パッケージ化する案）** — 不採用。両者の責務・データ契約・ライフサイクルの実証前に抽出すると、Switch Media側の未解決の欠陥（密結合・CSV出力の反映漏れ）を持ち込むリスクがあるため。

**影響範囲**: Architecture Review全体（§3「残す／再設計／廃止」の具体的マッピングはここで規定）。Master RoadmapのPhase構成（Phase 1が最小E2E実証、共通コア抽出はどのPhaseにも組み込まない＝実証後の条件付き判断として扱う）。

**再検討条件**: AMASとSwitch Media双方で、同一の責務・データ契約・ライフサイクル（例: 審査ルールエンジン、承認ワークフロー、監査ログ形式）が実際に稼働・検証された後に、共通コア抽出の可否を個別に再検討する。この再検討は自動的に発生せず、都度の合意を要する。

## D7｜Switch Media知見の利用

**決定済み**:
Switch Media側から現時点で取り込むのは、主に業務知識と設計パターン。

- 媒体別フィールド契約
- 審査ルールの外部化
- ルールベース→LLM差分の二段審査
- 部分成功・劣化継続
- SAFE／WARNING／NG
- 警告つき強行の承認UX
- Read-only→Write（PAUSED）→Financialの段階的権限
- common_id等による横断監査
- NGドメイン管理
- 媒体別文字数・入稿制約

Switch Mediaのコードを、そのままAMASの共通コアとしない。共通コア抽出の判断条件はD6を参照。

**決定理由**: Fable Review §11の転用5区分評価（そのままコード再利用可能／設計パターンのみ再利用可能／業務知識・ルールのみ再利用可能／AMAS向けに再設計が必要／Switch Media固有で転用しない）をそのまま採用。評価の結果、「そのままコード再利用可能」に該当したのはAMAS自身のAI生成基盤のみであり、Switch Media側の価値ある資産はすべて「設計パターン」または「業務知識・ルール」区分に分類されたため（Evidence §11）。

**不採用案**: Switch Mediaのコード（特にCSV出力ビルダー、reviewer.py等）をそのままAMASへ移植・共通化すること。理由: 「コードが似ているから共通化」は責務・データ契約・ライフサイクルの一致を検証していない判断であり、Fable Reviewが明示的に禁止した判定方法であるため（Evidence §11）。

**影響範囲**: Architecture Review §10（Switch Mediaとの関係）。Improvement Backlogに転用起因の新規設計課題があれば追加登録する。

**再検討条件**: D6と同一（両者の責務・データ契約・ライフサイクルの実証後）。

## D8｜最初の実証範囲

**決定済み**:
最初のEnd-to-End実証は以下に限定する。

- Google Ads
- 1キャンペーン
- 実媒体のテスト可能なアカウント
- 人間承認必須
- PAUSED状態までの入稿
- 実広告費を消化する配信開始は対象外
- 乱数、Mock、成功偽装を含めない
- 操作と結果を監査記録へ残す

目標ループ:
```
入力 → 生成 → 決定的バリデーション → AI審査 → 人間承認
→ Google AdsへPAUSED入稿 → 結果取得 → AMAS上の状態同期 → 監査記録
```

Financial権限と完全自律配信は、この実証の対象外。

**決定理由**: Fable Review §13案Dおよび§15（Validation Strategy）を採用。Fable Review §6が判定した「自律運用ループが観測・実行・結果取得・評価のすべてで断絶している」という現状（Evidence §6）に対し、まず1媒体・PAUSED入稿までの最小範囲でループを実際に閉じることで、以降のPhase設計・共通コア判断（D6）の材料を得る。

**不採用案**: 全媒体・自律配信までを一度に実装する案。理由: Fable Reviewが指摘する通り、現状のAMASには判断ルール・制約・実行権限・停止条件等の自律化基盤がほぼ全て欠落しており（Evidence §6）、範囲を広げるほど検証不能な複雑性が増すため。

**影響範囲**: Master Roadmap Phase 1がそのままD8の内容。Architecture Review §4（自律ループ）・§6（媒体Adapter＝Google Adsのみ先行）・§7（承認境界）。Security and Wallet Design（Financial権限は対象外のため、Phase 1範囲では予算消化ロジックの実装は不要）。

**再検討条件**: Phase 1のE2Eループが実データで1本通った時点（Production verified）で、Phase 2（Monitoring and Result Sync）以降の範囲をMaster Roadmapで確定する。
